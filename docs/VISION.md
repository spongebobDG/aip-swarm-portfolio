# AIP Swarm — 설계 철학 및 로드맵

---

## 핵심 목표

**동등한 자율 이동체들이 중앙 지휘 없이 협력하는 분산 군집 시스템.**

각 차량은:
- 독립적으로 자기 위치를 파악하고
- 독립적으로 임무를 수행할 수 있으며
- 다른 차량과 협력할 때 더 나은 성과를 낸다

단일 차량 고장이 전체 군집의 임무를 중단시키지 않는다.

---

## 현재와 목표의 괴리 — 예산 제약

이상적으로는 모든 차량이 동등한 사양이어야 한다.
그러나 **메인 AGV급 차량을 여러 대 동시 제작할 예산이 없기 때문에**
현재는 다음과 같은 임시 계층 구조가 존재한다.

```
목표 (동질 군집)              현재 (예산 제약 군집)
──────────────────────        ──────────────────────────────
[차량 A] ←→ [차량 B]          [메인 AGV]  ←→  [Scout × 2]
    ↕            ↕            (완전 사양)      (최소 사양)
[차량 C] ←→ [차량 D]          LiDAR, SLAM      ESP32, 센서 미정
                               독립 측위         중앙 의존 측위
```

### 용어 정의

| 코드 내 용어 | 실제 의미 | 임시 여부 |
|---|---|---|
| `main` | 완전 사양 피어 (1세대) | 임시 — 향후 모든 차량이 동급 |
| `scout_N` | 예산 제약 피어 (1세대) | 임시 — 사양 향상 시 `peer_N`으로 전환 |
| `supervisor` | 군집 상태 모니터 (중앙 PC) | 임시 — 장기적으로 각 차량에 분산 |
| `coordinator` | 추종 제어기 (중앙 PC) | 임시 — 장기적으로 각 차량 탑재 |

## 세대별 로드맵

### 1세대 (현재) — 예산 제약 이질 군집

```
중앙 PC (항상 켜져 있음)
  ├── supervisor: 군집 상태 모니터
  ├── watchdog: 장애 감지 → 전체 E-Stop
  ├── coordinator: 메인 기준 scout 추종 계산
  └── foxglove_bridge: 대시보드 게이트웨이

메인 AGV (완전 사양)
  ├── LiDAR + SLAM → 자기 위치 절대 파악
  ├── 카메라 → scout 위치 추정 보조
  └── cmd_vel 수신 (coordinator 발행)

Scout × 2 (최소 사양)
  ├── ESP32-S3: heartbeat, cmd_vel 수신
  ├── 위치: 메인 카메라에 의존 (ArUco 마커)
  └── 자체 SLAM 없음
```

**1세대 한계**:
- Scout의 위치 파악이 메인 카메라 시야에 의존
- 중앙 PC 다운 = 전체 군집 제어 불능
- Scout 독립 임무 수행 불가

---

### 2세대 (중기 목표) — 자기 위치 확보

각 Scout에 독립 측위 수단 추가. 중앙 PC는 여전히 존재하지만
각 차량이 자기 위치를 스스로 파악한다.

```
각 차량 (공통 최소 구성)
  ├── UWB 모듈 (차량 간 ranging)
  ├── 휠 엔코더 + IMU (dead reckoning)
  └── /<ns>/odom 독립 발행

위치 추정 방식
  ├── 1차: 자차 엔코더 + IMU 누적
  ├── 2차: UWB 차량 간 ranging으로 보정
  └── 3차 (메인만): LiDAR SLAM 절대 기준

중앙 PC 역할 축소
  ├── 대시보드 (Foxglove)
  └── 임무 할당 인터페이스
```

---

### 3세대 (장기 목표) — 완전 분산 군집

```
중앙 PC: 모니터링 + 임무 입력만 (없어도 군집 운용 가능)

각 차량 (동등 사양)
  ├── SLAM 또는 UWB 절대 측위
  ├── 로컬 플래너 + 충돌 회피
  ├── 이웃 차량 상태 구독 → 협력 계획
  ├── 임무 실패 시 다른 차량에 인계
  └── 군집 유지: 임의 차량 탈락해도 재편성

통신: 완전 P2P (FastDDS unicast, 중앙 DS 선택적)
보안: SROS2, 차량별 인증서
```

## 설계 원칙

### 1. 코드는 목표 아키텍처를 향해 쓴다

현재 중앙 PC에 있는 기능(coordinator, supervisor)은
**나중에 각 차량으로 이동할 것을 전제**로 작성한다.

```python
# 나쁜 예: 중앙 집중 의존
def get_all_vehicles():
    return HARDCODED_VEHICLE_LIST  # 중앙에서만 알 수 있음

# 좋은 예: 발견 기반
def get_online_vehicles():
    return self._heartbeat_cache.keys()  # 각 차량이 자신을 광고
```

### 2. 네임스페이스는 차량 정체성, 역할이 아니다

`/main/`, `/scout_1/` 네임스페이스는 **역할(role)이 아닌 정체성(identity)**.
`/main/cmd_vel` 토픽은 "메인이기 때문에 cmd_vel"이 아니라
"main이라는 이름의 차량의 cmd_vel".
모든 차량이 동일한 토픽 세트를 갖는다.

### 3. 중앙 PC는 단일 장애점이 아니어야 한다 (장기)

현재는 중앙 PC 없이 차량이 구동되지 않는다.
각 차량에 cmd_vel freshness watchdog(0.5 s)을 넣은 이유가 이것이다.
장기적으로 중앙 PC는 "있으면 편하지만 없어도 운용 가능"해야 한다.

### 4. 인터페이스는 사양에 무관하게 동일하다

ESP32 Scout와 Jetson 탑재 고사양 차량이 같은 토픽 스키마를 쓴다.
`FleetHeartbeat` 메시지는 두 경우 모두 동일하게 발행한다.
사양 차이는 내부 구현에서만 드러난다.

## 현재 코드의 임시 결정 목록

아래 항목은 예산 제약으로 인한 임시 결정이며,
사양이 올라가면 수정이 예상된다.

| 파일 | 임시 결정 | 제거 조건 |
|---|---|---|
| `supervisor_node.py` | vehicle_ids 정적 설정 | 차량 자동 발견으로 교체 |
| `coordinator_node.py` | 중앙 PC에 위치 | 각 차량 탑재 시 이동 |
| `scout_localizer_node.py` | 메인 카메라 기반 | 차량별 독립 측위 확보 시 선택적 사용 |
| `central.launch.py` | 중앙 PC에서 전체 실행 | 각 차량 launch 파일로 분산 |
| `docker-compose.yml` | 중앙 PC 컨테이너 스택 | 각 차량 컨테이너로 분산 |
| `supervisor.yaml` | vehicle_ids 하드코딩 | 동적 등록 프로토콜로 교체 |
