## 관점

이 문서는 **"각 차량이 어떻게 자신의 위치를 독립적으로 파악하는가"** 를 다룬다.

`scout_N`(예산 제약 피어)은 역할이 낮은 보조 차량이 아니라
**동일한 자율 군집 피어**이며, 현재의 사양 차이는 예산 제약에 의한 임시 상태다.
([VISION.md](VISION.md) 참조)

## 피어 사양 기준선 — 완전 사양 피어 (현 메인 AGV)

| 구성 요소 | 사양 |
|---|---|
| 컴퓨트 | Raspberry Pi 4B + Docker + ROS2 Humble |
| 측위 | YDLIDAR + IMU + 엔코더 → Full SLAM |
| 환경 인식 | ToF 범퍼 (충돌 방지) |
| 임무 탑재 | 3~4DOF 로봇암, RGB+열화상 퓨전, 가스/유해물질 센서 |
| 통신 | Wi-Fi (ROS2 DDS 네이티브) |

모든 예산 제약 피어의 업그레이드 목표는 이 사양이다.

---

## 현재 상태 (0단계) — 통신만 있는 최소 구성

```
ESP32-S3
  ├── micro-ROS → heartbeat 발행, cmd_vel 수신
  ├── 위치: 메인 AGV 카메라에 완전 의존 (ArUco 마커)
  └── 자체 판단 능력: 없음
```

**근본 한계**:
- 위치 파악이 완전 사양 피어의 카메라 시야에 의존
- 시야 이탈 = TF 끊김 = 정지
- 독립 임무 수행 불가
- 군집 피어 역할 수행 불가 (추종 객체에 가까움)

---

## 컴퓨팅 아키텍처 원칙

예산 제약 피어의 핵심 설계: **ESP32-S3를 안전 코프로세서로 유지하고
Raspberry Pi가 ROS2 인격체 역할을 담당한다.**

```
[Raspberry Pi (ROS2)]    UART/SPI    [ESP32-S3 (micro-ROS)]
  ───────────────────   ←────────   ───────────────────────
  /odom 발행                         모터 PWM 제어
  UWB 데이터 융합                    엔코더 인터럽트
  위치 추정 알고리즘                 하드 E-Stop 회로
  DDS 통신 (Wi-Fi)                   WDT 리셋
  임무 로직                          실시간 센서 인터럽트
```

이 구조의 장점:
- ESP32-S3는 하드 실시간 안전 기능에만 집중
- RPi는 ROS2 네이티브 실행 → micro-ROS Agent 의존성 제거 가능
- 향후 RPi만 업그레이드하면 컴퓨팅 성능 향상 (ESP32 재사용)

## 단계별 업그레이드 경로

### 1단계 — 자기 위치 인식 (즉시 착수, 약 9만 원)

**목표**: "어디 있는지는 스스로 안다"

```
[RPi Zero 2W] ──UART──▶ [ESP32-S3]
   ROS2 Humble              실시간 제어 (기존)
   UWB 융합                 엔코더 인터럽트
   위치 추정                하드 E-Stop
   /odom 발행
```

#### 추가 부품

| 부품 | 인터페이스 | 단가 | 역할 |
|---|---|---|---|
| **Raspberry Pi Zero 2W** | UART → ESP32 | ~$15 | ROS2 Humble 네이티브 실행 |
| **DWM3001C UWB** | I2C/SPI → RPi | ~$25 | 차량 간 거리 측정 (시야 무관) |
| **인크리멘탈 엔코더 × 2** | GPIO → ESP32 | ~$16 | 오도메트리 |
| **ICM-42688 IMU** | SPI → ESP32 | ~$8 | 헤딩·가속도 |
| **합계** | | **~$64 / 약 9만 원** | |

> UWB는 완전 사양 피어(메인 AGV)에도 1개 탑재 필요 (동일 단가).

#### 측위 원리 (UWB 차량 간 상호 ranging)

```
완전 사양 피어 (SLAM으로 자신의 절대 위치 known)
        │
        │ UWB ranging (시야 무관, 벽 통과)
        ▼
  예산 제약 피어 (거리 → 거리원)
        │
        │ IMU heading으로 방향 추정
        ▼
  map-frame 위치 추정 (20–30 cm 오차)
```

차량이 3대(메인 + 예산 피어 × 2)이면 교차 ranging으로 상호 보정:

```
A(known) ──range_ab──▶ B
A(known) ──range_ac──▶ C
B        ──range_bc──▶ C

→ B, C 절대 위치 추정 가능
```

#### 달성 능력

- UWB + 엔코더/IMU 융합 → 독립 위치 추정 (20–30 cm)
- 완전 사양 피어 카메라 없어도 자기 위치 파악
- `/<ns>/odom` 독립 발행 → coordinator 정상 동작
- 중앙 PC micro-ROS Agent 의존성 제거 가능

### 2단계 — 환경 인식 + 자율 회피 (약 +5만 원)

**목표**: "장애물을 스스로 피한다"

#### 추가 부품

| 부품 | 인터페이스 | 단가 | 역할 |
|---|---|---|---|
| **VL53L5CX × 4** (전·좌·우·후) | I2C → RPi | ~$28 | 근거리 장애물 감지 (최대 4 m) |
| **OV2640 카메라** (선택) | CSI → RPi | ~$10 | ArUco 직접 검출 or 시각 오도메트리 |
| **합계** | | **~$38 / 약 5만 원** | |

#### ToF 배열 구성

```
         [앞]
    [좌앞]   [우앞]      VL53L5CX × 4 (또는 6개로 확장)
  [좌]           [우]    각 8×8 = 64 depth points
    (선택: 좌뒤·우뒤)    → 256~384 포인트 / 프레임
```

| 항목 | YDLIDAR X2 | VL53L5CX × 4 |
|---|---|---|
| 각도 분해능 | 1° (360 포인트) | ~7.5° (성김) |
| 최대 거리 | 8 m | 4 m |
| 단가 | ~$80 | ~$28 |
| RPi Zero 2W 처리 | 가능 (ROS2 드라이버) | 가능 (I2C, 경량) |

#### 달성 능력

- 자율 장애물 회피 (충돌 방지)
- 카메라 추가 시 ArUco 직접 검출 → 완전 사양 피어 카메라 의존 완전 제거
- `/<ns>/scan` (LaserScan 변환) 발행 → Foxglove 3D 시각화

| 변경 | 단가 | 내용 |
|---|---|---|

- Full SLAM → 완전 사양 피어와 동등
- 중앙 PC coordinator 없이 독립 임무 수행 가능
- 군집 내 임의 차량 탈락 시 자동 재편성 가능

## ArUco 카메라 방식 병행 운용

`scout_localizer_node.py` (T7, 이미 구현)는 UWB 측위와 독립적으로 동작한다.
1단계 이후에도 완전 사양 피어의 카메라 TF를 **보조 보정 수단**으로 유지할 수 있다.

```
[완전 사양 피어 카메라] → ArUco → TF (절대, 시야 내, 주기적 보정)
[UWB ranging + IMU]    → 위치 융합 → TF (연속, 시야 무관)

→ 두 소스를 robot_localization EKF로 융합하면 정확도 최대
```

## ROS2 통합 구조 (목표)

```
[예산 제약 피어 — 1단계 이후]
  ├── /<ns>/odom          RPi: 엔코더+IMU dead reckoning
  ├── /<ns>/uwb_range     RPi: UWB ranging 원시값
  ├── /<ns>/scan          RPi: ToF → LaserScan 변환 (2단계~)
  └── map→<ns>/base_link  RPi: UWB 융합 TF 발행

[중앙 PC]
  ├── scout_localizer_node: 카메라 보조 보정 TF (선택)
  ├── uwb_localizer_node:   UWB 기반 TF (1단계, 추가 예정)
  └── coordinator_node:     추종 제어 (→ 3단계 이후 각 차량으로 이동)
```

UWB 노드는 `scout_localizer_node.py`와 독립적으로
`map→<ns>/base_link` TF를 발행하므로 `coordinator_node.py` 수정 없이 연동된다.

## 즉시 착수 체크리스트 (1단계)

하드웨어 준비 전 소프트웨어 선행 작업:

- [ ] `/<ns>/odom` 발행 노드 작성 (엔코더 + IMU 융합)
- [ ] UWB ranging → TF 발행 노드 작성 (`uwb_localizer_node.py`)
- [ ] `coordinator_node.py` fallback 로직: TF stale 시 odom으로 추정 연장
- [ ] `central.launch.py with_uwb_localizer:=true` 인자 추가
