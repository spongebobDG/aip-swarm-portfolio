# main+sub 통합본 실행 진단 기록 (2026-06-23)

## 현재 실행 결과

- 통합본 위치: `C:\Projects\aip-swarm-ws-main+sub`
- 웹 주소: `http://127.0.0.1:8080`

## 확인된 핵심 문제

현재 문제는 웹 카드 표시 문제가 아니라, 차량 상태를 중앙으로 가져오는 방식의 계약 불일치 문제다.

팀 main 폴더의 `FleetHeartbeat.msg`는 구형 계약이다.

```text
string<=32 vehicle_id
builtin_interfaces/Time stamp
uint8 state
float32 battery_pct
float32 cpu_load
string<=64[<=8] active_behaviors
```

반면 `scout_1` 실차 컨테이너가 실제로 발행하는 `/scout_1/heartbeat`는 신형 계약이다.

```text
std_msgs/Header header
string robot_id
string mode
bool healthy
bool estop
bool heartbeat_stale
bool obstacle_stop
bool cmd_stale
float32 battery_voltage
float32 battery_percentage
string status
```

ROS2/DDS에서는 토픽 이름이 같아도 메시지 정의가 다르면 서로 매칭되지 않는다. 즉 같은 이름인 `aip_fleet_msgs/msg/FleetHeartbeat`라도 필드가 다르면 중앙 supervisor가 해당 heartbeat를 받을 수 없다.

## 왜 팀 main에서는 aip1이 보였는데 통합본에서는 안 보이나

팀 main 프로젝트에서는 `aip1` heartbeat publisher와 supervisor가 모두 구형 `FleetHeartbeat.msg`를 기준으로 동작한다.

그래서 팀 main 기준에서는 `aip1`이 보일 수 있다.

이번 통합본은 `scout_1` 실차 heartbeat를 받기 위해 신형 `FleetHeartbeat.msg`로 맞췄다. 그러면 반대로 구형 heartbeat를 발행하는 `aip1`은 같은 프로세스에서 받을 수 없다.

요약하면 현재 상태는 다음과 같다.

| 차량 | 실제 heartbeat 계약 | 팀 main 구형 supervisor | 통합본 신형 supervisor |
|---|---|---:|---:|
| `aip1` | 구형으로 추정 | 받을 수 있음 | 받을 수 없음 |
| `scout_1` / 표시 `aip2` | 신형 확인됨 | 받을 수 없음 | 타입상 받을 수 있어야 함 |
| `scout_2` / 표시 `aip3` | 현재 heartbeat 미발행 (`USE_FLEET_ADAPTER=false`) | 받을 수 없음 | 받을 토픽 없음 |

## 합성이 필요한가

둘 중 하나를 선택해야 한다.

1. 팀 전체가 `FleetHeartbeat.msg`를 하나로 통일한다.
   - 가장 정석이다.
   - `aip1`, `scout_1`, `scout_2`, 중앙 supervisor가 모두 같은 메시지 정의를 써야 한다.
   - 단점: 팀 main 차량 SW 변경 승인이 필요하다.

2. 웹 관제 계층에서 표시용 상태를 합성한다.
   - 기존 차량 토픽/메시지 계약을 건드리지 않는 임시 통합안이다.
   - 예: 구형 heartbeat, 신형 heartbeat, odom/scan, UDP status side-channel을 각각 읽어서 웹 내부 표준 상태 `{id, online, mode, healthy, estop}`로 변환한다.
   - 장점: main 차량 원본 SW를 건드리지 않아도 된다.
   - 단점: 장기적으로는 임시 계층이 늘어나므로 최종 표준화가 필요하다.

현재 “main 폴더를 건드리지 말 것”이라는 조건에서는 2번, 즉 웹 관제 계층의 합성/어댑터 방식이 현실적이다.

## RTPS / tcpdump 확인

요청한 패킷 레벨 확인을 시도했다.

- `tcpdump` 설치 확인됨: `/usr/bin/tcpdump`
- 관제 PC WSL 인터페이스: `eth2`, IP `192.168.0.8/24`
- Discovery Server 포트: UDP `11811`
- 현재 WSL 사용자 권한으로 캡처 실패:

```text
tcpdump: eth2: You don't have permission to capture on that device
(socket: Operation not permitted)
```

따라서 현재 세션에서 RTPS 패킷 캡처는 sudo 권한 없이는 진행할 수 없다.

수동으로 실행할 명령은 다음과 같다.

```bash
sudo tcpdump -i eth2 -nn -vv udp port 11811
sudo tcpdump -i eth2 -nn -vv 'udp and portrange 7400-7600'
```

확인해야 할 포인트:

- 로봇 IP(`192.168.0.4`, `192.168.0.5`)에서 Discovery Server(`192.168.0.8:11811`)로 메타트래픽이 오는지
- 중앙 PC가 로봇 heartbeat publisher의 endpoint 정보를 받는지
- user-data locator가 실제 접근 가능한 `192.168.0.x` 주소인지, Docker/loopback/WSL 내부 주소로 광고되는지

- 웹 카드가 뜨는 것은 정상.
- 현재 차량 상태가 offline인 것은 웹 UI 문제가 아니라 heartbeat 계약/수신 경로 문제.
- `scout_2`는 우선 heartbeat publisher가 꺼져 있으므로, `USE_FLEET_ADAPTER=true`를 켤지 팀원 확인이 필요하다.
- `aip1`과 `scout_1`의 heartbeat 메시지 계약이 서로 달라, main 원본을 건드리지 않는 조건에서는 웹 관제 합성 계층이 필요하다.

## 팀 main 폴더의 기준 구조 확인

팀원이 만든 main 폴더에는 따라야 할 방향이 명확하게 들어 있다.

- 표준 차량 ID는 `aip1`, `aip2`, `aip3`이다.
- 구형 이름인 `main`, `scout_1`, `scout_2`는 폐기 대상으로 문서화되어 있다.
- 중앙 supervisor 설정도 `vehicle_ids: ["aip1", "aip2", "aip3"]`를 기준으로 한다.
- 핵심 토픽은 `/{vehicle_id}/heartbeat`, `/{vehicle_id}/scan`, `/{vehicle_id}/odom`, `/{vehicle_id}/cmd_vel`, `/{vehicle_id}/override_cmd_vel`, `/{vehicle_id}/estop`이다.
- 플릿 전역 상태는 `/fleet/status`를 기준으로 웹과 watchdog이 구독한다.
- 팀 main 기준의 `FleetHeartbeat.msg`는 `vehicle_id`, `stamp`, `state`, `battery_pct`, `cpu_load`, `active_behaviors` 필드를 가진 구형 계약이다.
- 팀 main 문서는 현재 네트워크를 `192.168.0.0/24`, `ROS_DOMAIN_ID=42`, FastDDS Simple Discovery 기준으로 설명한다. 지금 현장 통합본의 Discovery Server 방식과는 다르므로 이 차이는 별도로 정리해야 한다.

따라서 장기 방향은 웹에서만 이름을 바꾸는 것이 아니라, 실제 로봇 쪽 heartbeat와 토픽도 `aip1/aip2/aip3` 표준으로 맞추는 것이다. 단, 원본 main 폴더를 건드리지 않는 조건에서는 새 통합본에서만 임시 alias 또는 어댑터를 두고, 팀 승인 후 로봇 쪽 표준 토픽으로 옮기는 방식이 안전하다.
